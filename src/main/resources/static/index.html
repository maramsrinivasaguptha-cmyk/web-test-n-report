QA Scratchpad Scripting Guide

Write simple, line-by-line scripts to automate browser operations and verify Web UI components. Every command is executed sequentially inside a headless Chromium context. If any assertion fails or an error occurs, a ticket is created/updated and a failure screenshot is logged.

Navigation & Interaction

goto <URL>

Navigates the browser to the specified address.

goto https://example.com
click <Selector>

Clicks the element matching the selector.

click button#submit-form
fill <Selector> = <Value>

Clears and fills the text value into the input field.

fill input[name="q"] = Playwright Java
type <Selector> = <Value>

Types the characters key-by-key (useful for keyboard events).

type input#username = testuser
check <Selector> / uncheck <Selector>

Checks or unchecks a checkbox / radio option.

check input#agree-terms
press <Key>

Presses a keyboard key (e.g. Enter, Tab, Escape).

press Enter
wait <ms>

Pauses the execution for the given milliseconds.

wait 2000

Assertions

assert-text <Selector> = <Expected>

Asserts that the text content of the element contains the expected text. (Automatically waits for the selector to appear).

assert-text h1 = Example Domain
assert-title = <Expected>

Asserts that the page title contains the expected value.

assert-title = Google Search
assert-exists <Selector>

Asserts that the element matches at least one node in the DOM.

assert-exists .success-banner
assert-visible <Selector>

Asserts that the selector matches a visible element on the page.

assert-visible #logout-button

Best Practices & Selectors

  • Comments: Lines starting with # or // are ignored and serve as comments.
  • Selectors: Use standard CSS selector syntax:
    • ID: #element-id
    • Class: .button-primary
    • Attributes: input[type="email"]
    • Text content filters: text="Click Me" (Playwright specific)

E2E Pipelines & Advanced Testing

WebTestNReport supports fully chained end-to-end testing. You can define test stages, make API calls, query SQL databases, interface with simulated NoSQL stores, and pass variables between them.

API Testing & Variable Store

stage <Stage Name>

Declares a new visual stage in the test pipeline. Chained runs are structured by stage.

stage API Authentication
http-get / http-post / http-put / http-delete

Sends HTTP requests. Relative URLs beginning with / auto-complete to local server.

http-post /api/orders = {"qty": 5}
assert-status = <Code>

Asserts the HTTP status code of the last API response.

assert-status = 200
assert-json <Path> = <Value>

Asserts field value in last JSON response. Supports dot-notation.

assert-json user.id = 12
store-json <Path> = <VarName>

Extracts a JSON response field and stores it in variables context for subsequent steps.

store-json token = authToken

SQL & Simulated NoSQL Databases

db-connect [JDBC] [= User | Pass]

Connects to a database. Calling db-connect without arguments connects to local H2 database.

db-connect
db-query <SQL> / db-execute <SQL>

Runs SQL query or DDL/DML update statements.

db-query SELECT name FROM test_rules
assert-db <ColName> = <Value>

Asserts value in first row of last query result.

assert-db name = Check Example Domain
nosql-insert <Collection> = <JSON>

Inserts a document into simulated NoSQL document store (saved as JSON file locally).

nosql-insert carts = {"itemId": "item1"}
nosql-find <Collection> = <Query>

Queries documents matching filter object. Stores results in NoSQL context.

nosql-find carts = {"status": "pending"}
